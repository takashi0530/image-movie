# image-movie 本番リリース評価

> 「写真 → BGM 付きスライドショー動画」生成 Web アプリを一般公開する場合の、
> 必要対応・実現可能性・コスト・インフラ・パフォーマンス・収益化・需要の評価。

---

## 1. 実現可能性（結論：技術的には容易。事業的にはレッドオーシャン）

- 技術難易度は**低い**。コア（画像正規化 + ffmpeg エンコード）は実装・検証済みで動く。
- 差別化が難しい。後述のとおり競合が極めて多く、「ただの写真スライドショー」では勝てない。
- **勝ち筋は技術ではなく「ニッチ特化 × テンプレート × 日本語/国内決済」**にある。

---

## 2. 本番化に必要な対応（現状 → 本番のギャップ）

現状は「単一プロセス・インメモリのジョブ管理・ローカルディスク保存」。個人利用には十分だが、
マルチユーザー公開には以下が必須。

### 必須（これがないと公開できない）
| 項目 | 現状 | 本番で必要 |
|---|---|---|
| **BGM の権利処理** | `am.aac` の出所不明 | ⚠️ 最重要。著作権のある楽曲だと配布動画が違法に。ロイヤリティフリー音源（DOVA/Epidemic Sound 等）かライセンス取得、または「ユーザーが音源を持ち込む」設計に |
| ジョブ永続化 | インメモリ（再起動で消失） | キュー＋DB（Redis/Cloud Tasks + Postgres 等） |
| ファイル保存 | ローカルディスク | オブジェクトストレージ（S3 / GCS / R2） |
| ワーカー分離 | API と同居 | エンコードは専用ワーカーへ（API をブロックさせない） |
| 認証 | なし | ログイン（メール/SNS）。匿名利用も可だが課金には必須 |
| 濫用対策 | なし | レート制限、ファイルサイズ/枚数上限、画像の不適切コンテンツ検査、保存TTL |
| 監視/ログ | print のみ | 構造化ログ・エラー通知・メトリクス（ジョブ成功率/所要時間） |
| 法務 | なし | 利用規約・プライバシーポリシー（アップロード画像の取扱い明記）、特商法表記（課金時） |

### 望ましい（体験・運用品質）
- 進捗の％表示（ffmpeg の進捗パース）、生成失敗時の自動リトライ
- テンプレート（トランジション・尺・BGM・縦型/横型）
- 動画の自動削除（プライバシー & ストレージ削減）、ダウンロードURLの署名付き化

## 3. インフラ構成案（コスト/運用のバランス順）

エンコードは **CPU バウンド・状態あり** なので、純粋なステートレス FaaS には不向き。
「API は軽量・常時稼働、エンコードはキュー経由のワーカー」が基本形。

### 案A：マネージドPaaS（推奨・スモールスタート）
```
Frontend(Vercel) → API(Cloud Run) → Cloud Tasks → Worker(Cloud Run, CPU多め)
                                   ↘ 入出力: GCS（署名付きURL）
                                   ↘ ジョブ状態: Firestore or Cloud SQL
```
- 長所: スケール自動・運用最小。ワーカーは並列度を上げやすい。
- 短所: 1リクエスト最大時間やメモリに上限。大量画像/4Kは要チューニング。

### 案B：VPS + キュー（最安・自前運用）
- 1台の VPS（例: 月1,000〜数千円）に API + Redis + Celery ワーカー + ローカル/オブジェクトストレージ。
- 長所: 固定費が安く、ヘビーなエンコードもCPU次第で回せる。
- 短所: スケール・冗長化・運用は手動。バックアップ/監視を自前で。

### 案C：サーバレス（Lambda + SQS + S3）
- 長所: 完全従量・ゼロアイドルコスト。
- 短所: Lambda の15分制限・/tmp 容量・ffmpeg レイヤ管理。大きな動画でハマりやすい。

> 推奨: **まず案A（または案B）でMVP**。需要が見えてからワーカーのオートスケール最適化。

---

## 4. コスト感（marginal は安い、固定費と濫用が論点）

1本の生成（30枚・約75秒・1080p）あたり:
- **CPU**: 静止フレームの libx264 エンコードは軽く、1コアで概ね数秒〜十数秒 → **1本あたり 1円未満〜数円**。
- **ストレージ**: 1080p 75秒 ≈ 5〜15MB。TTLで自動削除すれば僅少。
- **転送(egress)**: ダウンロード 10MB ≈ 0.1〜0.2円。
- → **限界コストは非常に小さい**。問題は「無料濫用で固定費だけ膨らむ」こと。

固定費の目安:
- Vercel/Cloud Run の最小構成: **月 0〜数千円**から。
- 商用 BGM ライセンス（Epidemic Sound 等）: **月 数千〜2万円**規模（B2B配布可プランは要確認）。
- ドメイン・監視・決済手数料（Stripe 3.6%）など。

濫用対策（無料枠の制限・要ログイン・透かし）でコストは十分コントロール可能。

## 5. パフォーマンス方針

- **二重エンコードは既に廃止済み**（cv2+moviepy → ffmpeg 単一エンコード）。これが最大の改善。
- さらに伸ばすなら:
  - `-preset veryfast`〜`ultrafast` と `-crf` のバランスでエンコード時間↓（品質とのトレードオフ）。
  - 画像正規化を ffmpeg 側の `scale/pad` フィルタに寄せ、Python 前処理を削減（I/O 削減）。
  - 並列度はワーカー数で水平スケール（1本は1コアで十分速いので、本数をさばく設計に）。
  - 縦型(1080×1920)・SNS尺(15/30/60秒)プリセットで「重い4K」を避ける。
  - GPU エンコード(NVENC)は本数が非常に多い段階で検討（通常は不要）。
- 体感改善: 進捗％表示と「生成中も他操作可」の非同期UX（実装済みのジョブ方式と相性良）。

## 6. 収益化

| モデル | 内容 | 相性 |
|---|---|---|
| **フリーミアム + 透かし除去課金** | 無料は透かし/低画質、課金でHD・透かしなし | ◎ 王道。CV しやすい |
| 都度課金（pay-per-export） | 1本 ○円で書き出し | ○ ライトユーザー向け |
| サブスク | テンプレ・BGMライブラリ・本数無制限 | ○ ヘビーユーザー/法人 |
| B2B/受託 | 結婚式・葬儀・不動産・EC向けに特化提供 | ◎ 単価高・競合少 |
| 広告 | 無料層に広告 | △ 単価低、体験を損なう |

> 推奨: **フリーミアム（透かし）＋ HD書き出し課金** を軸に、ニッチB2B（後述）で単価を取る。
> BGM ライセンスのコストを賄えるかが損益分岐の鍵。

---

## 7. 需要と競合（正直な評価）

- **需要は大きいが市場は飽和**。汎用「写真→動画」は無料の強豪が多数:
  - CapCut / Canva / InShot / VivaVideo / Kapwing / Animoto
  - Google フォト「自動ムービー」・Apple「メモリー」（OS標準で無料）
- 汎用で正面勝負は不利。**差別化の方向**:
  1. **ニッチ特化**: 追悼/お別れ会スライド、結婚式プロフィールムービー、不動産物件紹介、EC商品動画、保育・学校行事 など「テンプレ＋納期＋安心」が効く領域。
  2. **国内最適化**: 日本語UI・国内決済（コンビニ/PayPay）・LINE連携・商用利用OK音源。
  3. **超高速・超簡単**: 「アップロードして30秒で完成」体験に全振り（多機能で勝たない）。
  4. **B2B 組み込み**: 写真館・葬儀社・不動産会社の業務フローにAPI/ホワイトラベル提供。

---

## 8. 推奨ロードマップ

1. **Phase 0（法務前提）**: BGM をライセンスクリアな音源に差し替え or 持ち込み式に。利用規約/プライバシー整備。
2. **Phase 1（MVP公開）**: 案A構成、認証、無料枠+透かし、TTL自動削除、進捗表示。1つのニッチに絞って訴求。
3. **Phase 2（収益化）**: HD書き出し課金（Stripe）、テンプレ/BGMライブラリ、SNS縦型対応。
4. **Phase 3（拡大）**: B2B/ホワイトラベル、ワーカーのオートスケール、品質/速度チューニング。

> 総括: **技術的実現性は高く、限界コストも小さい。最大の論点は「BGM 権利処理」と「飽和市場での差別化」。**
> 汎用ではなくニッチ特化で出すのが現実的な勝ち筋。
